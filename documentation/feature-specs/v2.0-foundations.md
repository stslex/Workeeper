## Locked decisions (carried in from planning)

These were resolved across three planning rounds and are now load-bearing for spec details below:

- **PR metric = heaviest set.** Weighted:`ORDER BY weight DESC, reps DESC, finished_at ASC LIMIT 1`.
  Weightless:`ORDER BY reps DESC, finished_at ASC LIMIT 1`. Tiebreak earliest (PR ownership goes to
  first time hit).
- **Chart default metric = same as PR.** Single aggregation pipeline serves both.
- **Ad-hoc trainings contribute** to the PR pool.
- **PR computed against finished sessions only.** In-progress sets count after finish, not before.
- **Edit past session triggers PR recompute.** On edit-save, recompute PRs for affected exercises (
  low frequency, acceptable cost).
- **Achievement block** — Variant C (three states: PR / Volume fallback / Onboarding). Rendering is
  v2.3; query support is v2.0.
- **Volume calculation** — only weighted exercises contribute. Weightless excluded from volume
  metric (mixing kg with raw reps is meaningless).
- **Active session conflict** — Variant C modal: **Resume** (default focus, primary), **Delete &
  start new** (destructive, outlined red), **Cancel** (text). Same-template tap → silent resume (no
  modal).
- **Live workout overflow Delete** — separate confirm dialog: **Cancel** (default focus) / **Delete
  ** (filled red). Asymmetric treatment vs. modal is intentional — overflow already gated
  tap-distance, dialog is the explicit confirmation.
- **Terminology** — "Delete" everywhere for hard removal of in-progress session. "Discard"rejected (
  collides with "Cancel" semantically).
- **Tag filter = OR** — keep current shipped behavior; AND-semantics query removal is intentional.

## Data layer

### New SessionDao queries

#### PR — heaviest set per exercise

```sql
SELECT s.weight, s.reps, sn.finished_at, sn.id AS session_id
FROM set s
JOIN performed_exercise pe ON s.performed_exercise_id = pe.id
JOIN session sn ON pe.session_id = sn.id
WHERE pe.exercise_id = :exerciseId
  AND sn.state = 'finished'
ORDER BY
  CASE WHEN :isWeightless = 0 THEN s.weight END DESC,
  s.reps DESC,
  sn.finished_at ASC,
  s.position ASC
LIMIT 1
```

Returns `null` when no finished sessions logged the exercise. The SQL `CASE` enforces the
weighted/weightless branch from a single Kotlin call site (
`getPersonalRecord(exerciseId, isWeightless)`).

Indexes used (already exist per `data-needs.md`):

- `performed_exercise(exercise_id)` — supports the `pe.exercise_id = ?` predicate.
- `session(state)` — supports the `state = 'finished'` predicate.
- `set(performed_exercise_id, position)` — supports the join + position tiebreak.

If profiling under realistic data (≥ 1000 sessions, ≥ 30 sets/session) shows this query exceeding ~
50 ms, add a partial composite index `set(performed_exercise_id, weight DESC, reps DESC)` — but only
as a follow-up; do not add prematurely.

#### Best session volume per training (and overall)

```sql
SELECT sn.id AS session_id, sn.training_id, sn.finished_at,
       SUM(s.weight * s.reps) AS volume
FROM session sn
JOIN performed_exercise pe ON pe.session_id = sn.id
JOIN exercise e ON e.id = pe.exercise_id
JOIN set s ON s.performed_exercise_id = pe.id
WHERE sn.state = 'finished'
  AND sn.finished_at >= :sinceMillis
  AND e.type = 'weighted'
GROUP BY sn.id
ORDER BY volume DESC
LIMIT :limit
```

Volume excludes weightless exercises per locked decision. Caller for v2.3 Achievement block uses
`:sinceMillis = now - 7 days, :limit = 1`. Stats dashboard (v2.3) uses wider windows.

#### History-by-exercise paged

```sql
SELECT sn.id AS session_id, sn.finished_at, sn.training_id, t.name AS training_name,
       t.is_adhoc, s.weight, s.reps, s.position, s.set_type
FROM set s
JOIN performed_exercise pe ON s.performed_exercise_id = pe.id
JOIN session sn ON pe.session_id = sn.id
JOIN training t ON sn.training_id = t.id
WHERE pe.exercise_id = :exerciseId
  AND sn.state = 'finished'
ORDER BY sn.finished_at DESC, s.position ASC
```

Wrapped as `PagingSource<Int, HistoryEntry>`. Already partially implemented in `feature/exercise`for
the recent-history section; this is a generalization with `Pager` semantics for v2.2.

### `SessionConflictResolver` — central helper

```kotlin
package io.github.stslex.workeeper.core.exercise.session

class SessionConflictResolver @Inject constructor(
    private val sessionRepository: SessionRepository,
) {
    suspend fun resolve(requestedTrainingUuid: String): Resolution {
        val active = sessionRepository.observeAnyActiveSession().firstOrNull()
            ?: return Resolution.ProceedFresh
        return when (active.trainingUuid) {
            requestedTrainingUuid -> Resolution.SilentResume(active.sessionUuid)
            else -> Resolution.NeedsUserChoice(active)
        }
    }

    sealed interface Resolution {
        data object ProceedFresh : Resolution
        data class SilentResume(val sessionUuid: String) : Resolution
        data class NeedsUserChoice(val active: ActiveSessionInfo) : Resolution
    }
}
```

Used by:

- `feature/home/.../HomeInteractor` — on training-picker tap.
- `feature/single-training/.../SingleTrainingInteractor` — on Start session.
- `feature/exercise/.../ExerciseInteractor` — on Track now.

Each caller, on `NeedsUserChoice`, surfaces the modal via
`Event.ShowActiveSessionConflict(activeSessionInfo)`. The composable graph reads the event and shows
`ActiveSessionConflictDialog` (defined in `core/ui/kit`).

### `finishSessionAtomic` — replacing `runCatching` + rollback

New `SessionRepository` method:

```kotlin
/**
 * Atomically finishes the session: applies plan updates per performed exercise, transitions
 * session state to `finished`, stamps `finished_at`. Single SQL transaction; failure rolls
 * back automatically without explicit compensating mutations.
 */
suspend fun finishSessionAtomic(
    sessionUuid: String,
    finishedAt: Long,
    planUpdates: List<PlanUpdate>,
): FinishOutcome
```

Implementation wraps the whole block in `database.withTransaction { ... }`.
`LiveWorkoutInteractorImpl.finishSession` becomes:

```kotlin
override suspend fun finishSession(sessionUuid: String): FinishResult? =
    withContext(defaultDispatcher) {
        val planUpdates = computePlanUpdates(sessionUuid)  // pure read, can stay outside the txn
        val outcome = sessionRepository.finishSessionAtomic(
            sessionUuid = sessionUuid,
            finishedAt = System.currentTimeMillis(),
            planUpdates = planUpdates,
        )
        outcome.toFinishResult()
    }
```

`runCatching` block and the `appliedMutations` rollback list disappear. If any mutation inside the
transaction throws, Room rolls back; the caller observes a thrown exception and surfaces an error
event upstream. The `appliedMutations.asReversed().forEach { runCatching { ... } }` anti-pattern (
silent failures during rollback) is gone.

### `recomputePersonalRecords(exerciseUuids: List<String>)` — new repository method

Triggered after edit-past-session. For each affected exercise, the materialized PR (if cached) is
invalidated. v2.0 ships **no PR cache** (the query is read-time); the method is a placeholder hook
for future caching, currently no-op or, alternatively, omitted entirely until v2.1. **Decision: omit
in v2.0**, add when caching is needed. Edit-past-session in v2.0 just relies on read-time queries to
reflect updated values.

(Sub-decision lock: "edit triggers recompute" is satisfied trivially when read-time queries are
used — every read recomputes.)

### ClickHandler — `processTrackNowClick` rewrite

```kotlin
private fun processTrackNowClick() {
    sendEvent(Event.Haptic(HapticFeedbackType.ContextClick))
    launch(defaultDispatcher) {
        val resolution = interactor.resolveTrackNowConflict(state.value.uuid)
        when (resolution) {
            is Resolution.ProceedFresh -> {
                val sessionUuid = interactor.startTrackNowSession(state.value.uuid)
                consumeOnMain(Action.Navigation.OpenLiveWorkout(sessionUuid))
            }
            is Resolution.SilentResume -> {
                consumeOnMain(Action.Navigation.OpenLiveWorkout(resolution.sessionUuid))
            }
            is Resolution.NeedsUserChoice -> {
                consumeOnMain(Action.Common.ShowConflict(resolution.active))
            }
        }
    }
}
```

`startTrackNowSession` creates an ad-hoc training with one exercise (current screen's exercise),

`consume(Navigation.OpenLiveWorkout(activeSessionInfo.sessionUuid))`.

`Click.OnTrackNowDeleteAndStart` →
`interactor.deleteSession(activeSessionInfo.sessionUuid); processTrackNowClick()` (recursive, but
with active now cleared, falls into ProceedFresh branch).

`Click.OnTrackNowConflictDismiss` → no-op.

```
                  └─> DiscardSessionConfirmDialog rendered
                       ├─ Delete → interactor.deleteSession, Navigation.Back
                       └─ Cancel → updateState(deleteDialogVisible = false)
```
