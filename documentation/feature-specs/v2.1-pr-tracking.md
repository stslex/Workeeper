# Feature spec — v2.1 PR tracking (Stage v2.1)

**Status:** Implemented. Built on v2.0 foundations (merged in PR #75). See PR for v2.1 changes.

This stage ships personal-record (PR) tracking across three user-facing surfaces. Architecture choice: **Room native `Flow` everywhere**, no manual cache or invalidation layer. Room auto-invalidates queries when their participating tables change, so finishing a session, editing a past session, deleting a session, or permanently deleting an exercise re-emits PR values to all subscribers without explicit plumbing.

## Architectural notes

### Reactive layer = Room native `Flow`

Existing project pattern (verified in `TagRepository`, `TrainingRepository`): DAO method returns `Flow<...>`, repo wraps with `.map { ... }.flowOn(ioDispatcher)`. Room recompiles the query and re-emits whenever any of the involved tables changes.

For PR (`SELECT ... LIMIT 1` over indexed columns), the cost of an irrelevant re-execution is negligible — a few milliseconds at most. We do **not** add manual invalidation, custom event bus, `MutableStateFlow` caches, or per-key state maps. The repo stays stateless; the data layer is pure pass-through over Room.

### What this means for each surface

- **Exercise detail and Past session** subscribe to the PR `Flow`. When live workout finishes, those screens (if backstacked) auto-update via Room invalidation. Edit-past-session save → same.
- **Live workout uses `firstOrNull()` once on session load** to freeze a snapshot. The snapshot does not update mid-session even though the underlying flow would emit on every set save — Q6 lock requires session-frozen comparator semantics.

### Caching policy

For v2.1 surfaces (PR queries are light), no caching layer.

For v2.2 (chart series — heavier compute over potentially hundreds of rows), the question of whether to cache aggregations becomes real. **Out of scope for v2.1.** Tech-debt entry added below to flag the horizon decision; do not preempt it with speculative APIs.

## Scope

Three consumer surfaces for PR data, plus a small in-moment detection helper. All read-time, no caching.

### 1. Exercise detail — PR card (read mode)

A new card rendered between `DefaultPlanCard` (when present) and `HistorySection`'s `Recent` heading. Per Q1 placement decision (mockup option C).

Visibility: `state.personalRecord != null`. When the exercise has zero finished-session sets, the card is hidden.

Card layout (mockup option C styling): amber accent band + `"PR"` badge, body line 1 = pre-formatted display label, body line 2 = relative date label. Tap behavior: **none in v2.1** (chart entry point lands in v2.2).

The card subscribes to `PersonalRecordRepository.observePersonalRecord(uuid, type)`; when a new session finishes that bumps the PR, the card auto-refreshes.

### 2. Live workout — in-moment PR highlight

Set rows in `LiveExerciseCard` matching a PR get the **left-accent-strip + badge** treatment (mockup option C): 3dp amber left-border, `"PR"` pill on the right.

Detection (per Q6 lock — pre-session snapshot scope):

- At session load (`LiveWorkoutInteractor.loadSession`), call `personalRecordRepository.observePersonalRecords(uuidsByType).firstOrNull()` to take a one-shot snapshot. The result is stored as immutable `Map<String, PrSnapshotItem?>` in `LiveWorkoutStore.State`. **The flow is not held open** — we explicitly want session-frozen behavior; mid-session updates from other places must not flip in-moment highlights.
- A logged set is a PR iff it strictly beats the snapshot value:
  - **Weighted** (`type == WEIGHTED`): `(set.weight ?: 0.0) > (snapshot.weight ?: 0.0)` OR `(set.weight == snapshot.weight && set.reps > snapshot.reps)`.
  - **Weightless** (`type == WEIGHTLESS`): `set.reps > snapshot.reps`.
  - **No snapshot** (`null`): any logged set with non-null weight (weighted) or `reps > 0` (weightless) is a PR.

No snackbar, no animation. Pure visual marker.

### 3. Live workout finish dialog — "New personal records" block

`LiveWorkoutStore.State.FinishStats` extends with:

```kotlin
data class FinishStats(
    // ... existing
    val newPersonalRecords: ImmutableList<NewPrEntry>,
) {
    @Stable
    data class NewPrEntry(
        val exerciseUuid: String,
        val exerciseName: String,
        val displayLabel: String,    // pre-formatted "105 × 5" / "15 reps"
    )
}
```

Renders below existing stat rows in `FinishConfirmDialog` when non-empty: amber-tinted card with header `"X новых рекордов"` + one row per exercise.

If the same exercise bit the PR multiple times in the session (3 successive heavier sets), only the **best one** appears. Computation walks `state.exercises[*].performedSets`, picks the best per exercise, compares against `state.preSessionPrSnapshot`. No DB hit at finish time.

### 4. Past session detail — PR badge on set rows

`PastSetUiModel` gains `isPersonalRecord: Boolean`. Renders with the same accent + badge treatment as Live workout.

Detection: at past-session load, the interactor subscribes to `observePersonalRecords(uuidsByType)`. For each set, `isPersonalRecord = (currentPr.setUuid == set.setUuid)`. Only **one set per exercise** can carry the badge (the latest best, which is what's currently the PR).

When the screen edits a set and saves, the underlying `set_table` change triggers Room re-execution; Past session receives the updated PR map and updates badges. No manual recompute call needed.

## Locked decisions

Recap from planning rounds:

- **PR metric = heaviest set.** Weighted: `weight DESC, reps DESC, finished_at ASC LIMIT 1`. Weightless: `reps DESC, finished_at ASC LIMIT 1`. SQL already encoded in v2.0.
- **Ad-hoc trainings contribute** to the PR pool.
- **PR = finished sessions only.** Snapshot taken at session start; in-progress sets compared against it but don't enter the pool until finish.
- **Edit past session triggers PR recompute.** Naturally satisfied via Room Flow re-emission.
- **Q1** — PR card placement = above `HistorySection`.
- **Q2** — In-moment highlight = left-accent-strip + badge.
- **Q3** — Session summary = extend finish confirm dialog.
- **Q4** — Past session PR badge = yes, on rows that are the current PR.
- **Q5** — No PR cache. Read-time queries via Room Flow. Per-session snapshot in `LiveWorkoutStore.State` is **session-scoped immutable data**, not a cache.
- **Q6** — In-moment scope = pre-session snapshot. Snapshot frozen at session load; never updated mid-session.

## Data layer — DAO + repo extensions

### `SessionDao.observePersonalRecord` (new)

Same SQL body as the existing `getPersonalRecord` (shipped in v2.0), only the return type differs:

```kotlin
@Query(
    """
    SELECT s.uuid AS set_uuid,
           s.weight AS weight,
           s.reps AS reps,
           s.type AS type,
           s.performed_exercise_uuid AS performed_exercise_uuid,
           sn.uuid AS session_uuid,
           sn.finished_at AS finished_at
    FROM set_table s
    JOIN performed_exercise_table pe ON pe.uuid = s.performed_exercise_uuid
    JOIN session_table sn ON sn.uuid = pe.session_uuid
    WHERE pe.exercise_uuid = :exerciseUuid
      AND sn.state = 'FINISHED'
      AND sn.finished_at IS NOT NULL
      AND (:isWeightless = 1 OR s.weight IS NOT NULL)
    ORDER BY
        CASE WHEN :isWeightless = 0 THEN s.weight END DESC,
        s.reps DESC,
        sn.finished_at ASC,
        s.position ASC
    LIMIT 1
    """,
)
fun observePersonalRecord(exerciseUuid: Uuid, isWeightless: Boolean): Flow<PersonalRecordRow?>
```

Existing `suspend` variant (`getPersonalRecord`) is kept for one-off uses.

### `PersonalRecordRepository` extensions

```kotlin
interface PersonalRecordRepository {

    /** existing one-shot, kept for one-off lookups. */
    suspend fun getPersonalRecord(exerciseUuid: String, type: ExerciseTypeDataModel): PersonalRecordDataModel?

    /**
     * Reactive PR for a single exercise. Re-emits when finished-session sets for this
     * exercise change (Room handles invalidation). Subscribe in screens that should
     * stay live-updated; for one-shot freezes (e.g. Live workout pre-session snapshot)
     * use [observePersonalRecords] + `firstOrNull()` or stick to [getPersonalRecord].
     */
    fun observePersonalRecord(
        exerciseUuid: String,
        type: ExerciseTypeDataModel,
    ): Flow<PersonalRecordDataModel?>

    /**
     * Reactive PR map for a batch of exercises. Implemented via [combine] over per-exercise
     * flows; downstream gets a fresh map each time any contributor emits. The map's keyset
     * matches [uuidsByType] exactly; values are nullable when the exercise has no PR yet.
     */
    fun observePersonalRecords(
        uuidsByType: Map<String, ExerciseTypeDataModel>,
    ): Flow<Map<String, PersonalRecordDataModel?>>
}
```

`observePersonalRecords` impl — straightforward `combine`:

```kotlin
override fun observePersonalRecords(
    uuidsByType: Map<String, ExerciseTypeDataModel>,
): Flow<Map<String, PersonalRecordDataModel?>> {
    if (uuidsByType.isEmpty()) return flowOf(emptyMap())
    val perExerciseFlows: List<Flow<Pair<String, PersonalRecordDataModel?>>> =
        uuidsByType.map { (uuid, type) ->
            observePersonalRecord(uuid, type).map { uuid to it }
        }
    return combine(perExerciseFlows) { pairs -> pairs.toMap() }
}
```

This is the project's first `combine` callsite (verified — currently zero usages); the import is plain `kotlinx.coroutines.flow.combine`.

### `PrComparator.kt` — pure-function utility

Lives at `core/exercise/.../sets/` (next to `PlanUpdateRule`):

```kotlin
internal object PrComparator {

    /**
     * True if [candidate] strictly beats [baseline] for an exercise of [type].
     * `null` baseline means no PR exists yet — any non-zero candidate beats it.
     */
    fun beats(
        candidate: PlanSetDataModel,
        baseline: PersonalRecordDataModel?,
        type: ExerciseTypeDataModel,
    ): Boolean

    /**
     * Picks the best set in [sets] for an exercise of [type]. Returns null if [sets] is empty.
     * "Best" follows the same ordering as the SQL PR query.
     */
    fun bestOf(
        sets: List<PlanSetDataModel>,
        type: ExerciseTypeDataModel,
    ): PlanSetDataModel?
}
```

Used by:
- `LiveWorkoutMapper.toLiveExerciseUiModel` to mark each `LiveSetUiModel.isPersonalRecord` (calls `beats`).
- `LiveWorkoutMapper.toFinishStats` to compute `newPersonalRecords` (calls `bestOf` per exercise, then `beats` against snapshot).

The Kotlin `PrComparator` and the SQL `ORDER BY` in `observePersonalRecord` are two parallel implementations of the same comparator. Parity is asserted by `PrComparatorTest`: for a fixed dataset seeded into a Room in-memory DB, `bestOf(...)` returns the same set the DAO selects.

Past session **does not** use `PrComparator`. It uses raw setUuid equality against the current PR (which Room already determined). Set-uuid match is cheaper than re-running the comparator.

### Interactor change

Replace existing one-shot detail loader with a flow that combines detail + PR map:

```kotlin
fun observeDetailWithPrs(
    sessionUuid: String,
): Flow<DetailWithPrs?>

data class DetailWithPrs(
    val detail: SessionDetailDataModel,
    val prMap: Map<String, PersonalRecordDataModel?>,
)
```

Impl:

```kotlin
override fun observeDetailWithPrs(sessionUuid: String): Flow<DetailWithPrs?> = flow {
    val detail = sessionRepository.getSessionDetail(sessionUuid)
    if (detail == null) {
        emit(null)
        return@flow
    }
    val uuidsByType: Map<String, ExerciseTypeDataModel> = detail.exercises.associate { it.exerciseUuid to it.type }
    emitAll(
        personalRecordRepository
            .observePersonalRecords(uuidsByType)
            .map { prMap -> DetailWithPrs(detail, prMap) }
    )
}.flowOn(ioDispatcher)
```

A subtle point: this collapses two reads (session detail one-shot + PR observe) into one `Flow`. Session detail itself does not auto-update via Room here — the existing `getSessionDetail` is `suspend`. If editing a set updates the detail row's content (e.g. user changes weight 100→105), the **PR** flow re-emits and the set's `isPersonalRecord` flag updates, but the **set value** displayed in the row (which comes from `detail`) does not auto-refresh.

In the **current shipped** Past session, edits are reflected via mapper-driven local state mutation in handler — the screen optimistically updates state on save without re-fetching. This is sufficient for v2.1: user edit → optimistic state update → save commits → Room invalidates → PR map re-emits → `isPersonalRecord` refreshes. The set values themselves stay consistent because the optimistic update wrote them.

**Verify during implementation:** confirm the existing edit-save flow is optimistic. If it does a re-fetch instead, switch this Flow to also re-fetch detail on PR re-emission — slightly heavier but more consistent.

## Tech-debt register updates

### Closed by this stage

None.

### New entries (added by this spec)

| Severity | Location | Description |
|---|---|---|
| 🟡 PARKED | TBD per v2.2 stage | **Heavy-aggregation re-execution policy.** Room native Flow re-runs queries on every change to involved tables, regardless of whether the change is relevant to the specific query. For PR (LIMIT 1, indexed) the cost is negligible; v2.1 ships as-is. For v2.2 chart series (full history scan + Kotlin-side bucketing), the cost may matter if a session logs many sets while a chart is observed. **Trigger to act:** profiling in v2.2. **Anti-patterns to avoid:** repo-level `mutableMapOf<Key, StateFlow>` caches (subscription leaks, concurrent mutation, no clean lifecycle owner — entries removed from the map don't cancel upstream subscriptions; collectors cancelled on the consumer side leave the StateFlow producer alive); manual event-bus invalidators built on top of Room (duplicates a primitive Room already provides). **If a cache is needed,** put it at the consumer side (`stateIn(viewModelScope, WhileSubscribed(...), initial)` on the interactor) where lifecycle is bounded and the StateFlow disappears with the consumer. |
