## 1. Scope

### In scope

- New "Start blank" entry in the Home `StartWorkoutWidget` picker, alongside
  existing library trainings.
- Live workout screen extended to support an empty initial state (no
  `TrainingExerciseEntity` rows, no exercises picked).
- Editable training name header in Live workout (save on blur, "Untitled"
  placeholder when empty).
- Inline exercise picker (bottom sheet) with search-or-create UX. Explicit
  no-match indicator before the "Create '{query}'" CTA.
- **Mid-session add exercise capability for all sessions** — applies to
  ad-hoc Quick start sessions, ad-hoc Track Now sessions, and library
  training sessions. Mid-session additions mutate the plan permanently (rule
  H1).
- Empty-finish confirm dialog: Discard or Continue editing. No save-anyway
  option.
- `ExerciseEntity.isAdhoc` boolean flag with full create/graduate/delete
  lifecycle.
- Migration v5 → v6 adding the `is_adhoc` column to `exercise_table` and
  retroactively cleaning orphan ad-hoc `TrainingEntity` rows from prior Track
  Now cancel flows.
- Refactor of Track Now cancel path to use the new `discardAdhocSession`
  interactor helper (fixes prior orphan Training bug as a side effect).
- Shared `createAdhocSession(name, exerciseUuids)` interactor helper used by
  both Track Now and Quick start entry points.

### Non-goals (explicitly deferred)

- **Library training session add-exercise UI parity is in scope, but
  per-session vs per-plan differentiation is not.** Mid-session adds always
  mutate the plan permanently. A "today only" toggle is deferred to a future
  stage if regretted.
- **Track Now / Quick start UI unification.** They share the data layer
  (`createAdhocSession`, `discardAdhocSession`) but remain separate UI flows.
  Refactor candidate for v2.7.
- **Abandoned ad-hoc session cleanup beyond the v5→v6 migration sweep.**
  `IN_PROGRESS` parallel-session creation is already blocked at the UI level;
  the existing Home banner remains the resume/cancel surface. No periodic
  cleanup job.
- **Inline exercise editor enrichment** (tags, default body part, type
  selection during inline create). Inline create captures **only the name**;
  type defaults to `WEIGHTED` (current default per `ExerciseStore.kt:146`);
  user can enrich later from Exercise detail.
- **Auto-cleanup of zero-set exercises on finish.** If user added an exercise
  mid-session and logged 0 sets, the `TrainingExerciseEntity` row stays in
  the plan (per H1.b — "user does not lose what they create").

### H1 implicit consequences (confirmed)

| ID | Consequence |
|----|-------------|
| H1.a | Newly-added mid-session exercise's `TrainingExerciseEntity.planSets` starts as `null`; existing `PlanUpdateRule` (grow-but-not-shrink) populates it on finish from performed sets. |
| H1.b | If exercise is added mid-session but 0 sets logged, the `TrainingExerciseEntity` row is kept on finish (no auto-cleanup). User can remove it later via plan edit UI. |
| H1.c | Cancel session for any ad-hoc training cascades: delete session AND delete the ad-hoc training row AND delete inline-created (`is_adhoc=true`) exercise rows associated with it. |
| H1.d | Discard from empty-finish dialog uses the same cascade path as Cancel session. Implemented via shared `discardAdhocSession` helper, two call sites in handler. |

### Adhoc lifecycle (`ExerciseEntity.isAdhoc`)

- **Create**: Inline exercise creation in mid-session picker → row inserted
  with `is_adhoc = true`. A case-insensitive name collision with any
  existing exercise row (via DAO `findByName`) returns the existing row
  with `reusedExisting = true` instead of creating a duplicate; the
  existing row's `is_adhoc` flag is **not** modified.
- **Graduate**: On session finish, the ad-hoc training row and all exercises
  associated with the finishing training graduate to `is_adhoc = false`
  regardless of sets logged. After graduate, the training and exercises are
  indistinguishable from natively created library rows.
- **Delete (defence in depth)**: On cancel/discard of ad-hoc training, only
  exercises matching **both** predicates are deleted: `is_adhoc = true`
  **AND** present in `TrainingExerciseEntity` join for the cancelled
  training. Library exercises (even those just picked from the picker into
  this session) cannot be accidentally deleted because their `is_adhoc` is
  already `false`.
- **Picker filter**: Exercise picker queries (and any future exercise list
  surface) filter `is_adhoc = false`. Inline-created exercises in the
  current session are not shown in the picker — they're already in the
  active session list.

**Name collision handling**: tapping the Create CTA invokes
`createInlineAdhocExercise(name)`. The repository performs a
case-insensitive `findByName` lookup before insert (the DAO query uses
`COLLATE NOCASE`). If a match exists — whether `is_adhoc = 0` library
row or `is_adhoc = 1` orphan row from a prior crashed session — the
existing row is returned with `reusedExisting = true`. No new row is
inserted, no field on the existing row (including `is_adhoc`) is
mutated. The picker handler then fetches the PR baseline for that
exercise (since it has potential history). If no match exists, a fresh
`is_adhoc = true` row is inserted with `reusedExisting = false` and the
PR fetch is skipped (no baseline to fetch). Lookup + insert run inside
a single `transition { }` block so a concurrent insert with the same
name cannot race past the lookup.

`exercises.all { it.performedSets.isEmpty() }`.

```
┌──────────────────────────────────────────────┐
│        Discard empty workout?                │
├──────────────────────────────────────────────┤
│  This workout has no logged sets. It looks   │
│  like you haven't started yet.               │
│                                              │
│        [ Continue editing ]                  │
│        [ Discard ]                           │
└──────────────────────────────────────────────┘
```

- Continue editing → dismiss dialog, no state change.
- Discard → `Action.Click.OnEmptyFinishDiscard` →
  `interactor.discardAdhocSession(sessionUuid, trainingUuid)` → on success
  emit `Action.Navigation.PopBack`.

Discard CTA is enabled **only** for ad-hoc trainings (`isAdhoc = true`). For
library training sessions, the empty-finish path uses a different dialog —
"Continue editing" only, no Discard option (we don't delete library
trainings via session cancellation). This branch is internal to the handler;
spec lock is "discard option exists iff training.isAdhoc".

### 4.6 Finish dialog — name required (Q2)

When user taps Finish on a non-empty session and `training.name.isBlank()`,
the existing finish dialog (already extended in v2.1 for PR summary) gains a
required name field at the top. Submit blocked until name is non-empty. No
"save without name" path.

On successful submit, finish runs as one transactional persistence step:
session state flips to `FINISHED`, plan updates are applied, linked exercises
graduate to `is_adhoc = false`, and the ad-hoc training row also graduates to
`is_adhoc = false`. A failed finish cannot preserve only one side of that
graduation.

**Atomicity (rename + finish)**: when the dialog supplies a new training
name, the rename is applied inside the same `transition { }` block as the
state flip + graduation via `finishSessionAtomic(newTrainingName = ...)`.
A throw at any point in the block rolls back the rename together with the
state flip, so a crash between writes can never leave a named-but-IN_PROGRESS
session or an unfinishable named training. The standalone
`updateTrainingName` interactor method is reserved for the header-blur edit
path, where rename is independent of any session-state change.

## 6. PR snapshot extension (C1 — lazy fetch + merge)

### Problem

Current behavior (v2.1): `LiveWorkoutInteractorImpl.kt:81-87` collects PR
snapshot once via `firstOrNull()` at session start. The snapshot is a frozen
`Map<String, PrSnapshotItem>`. Mid-session-added exercises have no entry.

### Solution

Snapshot becomes incrementally populated. The State map is **merged**, never
replaced.

#### Flow

1. Session start: existing logic fetches snapshot for all initially loaded
   exercises (zero entries for blank init). Stored in State.
2. Mid-session add of **library exercise** (picked, has history):
   - Handler dispatches `addExerciseToActiveSession`.
   - In parallel: `personalRecordRepository.fetchPrSnapshotForExercise(
     exerciseUuid, exerciseType)` — single suspend call returning
     `PrSnapshotItem?`.
   - On result: `updateState { state.copy(preSessionPrSnapshot =
     state.preSessionPrSnapshot + (exerciseUuid to snapshot)) }` — **map
     plus**, not replace.
3. Mid-session create of **inline ad-hoc exercise** (no history):
   - Handler creates exercise + adds to session.
   - **No fetch** — empty history, no baseline.
   - State map unchanged for this exercise key.
   - In-moment PR detection rule (Q6): `preSessionPrSnapshot[uuid] == null`
     → `hasBaseline = false` → in-moment badge **suppressed**.
   - On finish, session summary surfaces a single "Established baseline for
     {exercise}" entry per such exercise (existing summary infrastructure
     extended).

#### Concurrency — throttle + merge

User can tap "Add exercise" in rapid succession. Two parallel suspend
fetches for two different exercises are **safe** because the State update
uses map-plus semantics — order of resolution doesn't matter, both keys end
up in the map.

Throttle is added on the UX side: `canAddExercise = !isAddExerciseInFlight
&& !isFinishInFlight`. The picker bottom sheet's primary CTA is disabled
during in-flight fetch. This prevents both UX confusion (rapid double-add)
and unnecessary parallel work.

#### Failure handling

If `fetchPrSnapshotForExercise` fails, the exercise is still added to the
session (session add and PR fetch are independent operations). Snapshot for
that exercise stays absent from the map → in-moment badge suppressed for
this exercise → identical behavior to inline-created exercise. User sees
exercise added; no error toast (PR is a nice-to-have, not blocker). This is
a deliberate degraded-mode choice — flag in tech-debt if telemetry shows
this firing often.

```kotlin
suspend fun discardAdhocSession(

    val adhocExerciseUuids = exerciseDao
        .getAdhocExercisesForTraining(trainingUuid.toUuid())
        .map { it.uuid }

    sessionDao.deleteSession(sessionUuid.toUuid())
    trainingDao.deleteTraining(trainingUuid.toUuid())
    // training_exercise_table rows cascade via FK on training_uuid
    if (adhocExerciseUuids.isNotEmpty()) {
        exerciseDao.deleteExercises(adhocExerciseUuids)
    }
}
```

The transaction guarantees atomicity. The defence-in-depth predicate
(`is_adhoc = 1` from the join query) ensures library exercises picked into
this session are never deleted — their `isAdhoc` is `false` so they're
filtered out at the join step.

Finish graduation also runs in the same Room transaction as the session finish:

```kotlin
@Query("""
    UPDATE exercise_table SET is_adhoc = 0
    WHERE uuid IN (
        SELECT te.exercise_uuid FROM training_exercise_table te
        WHERE te.training_uuid = :trainingUuid
    )
""")
suspend fun graduateAdhocForTraining(trainingUuid: Uuid)

@Query("UPDATE training_table SET is_adhoc = 0 WHERE uuid = :trainingUuid")
suspend fun graduateTraining(trainingUuid: Uuid)
```

## 8. Migration v5 → v6 (non-destructive)

```kotlin
internal object Migration6 : Migration(5, 6) {
    override fun migrate(db: SupportSQLiteDatabase) {
        // 1. Add the new column with safe default.
        db.execSQL("""
            ALTER TABLE exercise_table
            ADD COLUMN is_adhoc INTEGER NOT NULL DEFAULT 0
        """.trimIndent())

        // 2. Retroactively clean orphan ad-hoc training rows from prior
        //    Track Now cancel flows. Sessions referencing them are already
        //    deleted; the training rows leaked.
        db.execSQL("""
            DELETE FROM training_table
            WHERE is_adhoc = 1
              AND uuid NOT IN (SELECT training_uuid FROM session_table)
        """.trimIndent())

        // 3. training_exercise_table rows referencing deleted trainings
        //    cascade via existing FK on training_uuid. No explicit cleanup
        //    needed.
    }
}
```

Register in `CoreDatabaseModule.kt` migrations list. Schema version bumped
to `6` in `AppDatabase.kt`.

### Migration test

`Migration6Test` (instrumented Room migration test) verifies:

1. Pre-migration data with `is_adhoc` absent → post-migration has column
   with all rows defaulted to `0` (false).
2. Pre-migration orphan ad-hoc training row (no session referencing it) →
   post-migration the row is deleted.
3. Pre-migration valid ad-hoc training row (session referencing it exists)
   → post-migration the row is preserved.
4. Pre-migration library training row (`is_adhoc = 0`) with no session
   referencing it → post-migration the row is preserved (only ad-hoc
   orphans are swept).
