# Codebase research — v2.3 "Quick start workout" feature

This document provides a technical research summary of the existing Workeeper codebase to inform the specification of the "Quick start workout" feature (v2.3).

## 1. Live workout initialization flow

**How is a Live workout session currently started?**
A fresh Live workout session is started from the Home screen via a Training picker. The flow is:
1.  **Home Entry Point**: `HomeScreen.kt:143-157` dispatches `Action.Click.OnStartTrainingClick`.
2.  **Picker Selection**: `ClickHandler.kt:94-98` handles training selection and dispatches `Action.Navigation.OpenLiveWorkoutFresh(trainingUuid)`.
3.  **Navigation**: `NavigationHandler.kt:40` in `feature/home` maps this to `Screen.LiveWorkout(sessionUuid = null, trainingUuid = action.trainingUuid)`.
4.  **Processor Init**: `LiveWorkoutStoreImpl.kt:44-50` sends `Action.Common.Init` on initialization.
5.  **Session Creation**: `CommonHandler.kt:80-92` in `feature/live-workout` calls `createSession(current.trainingUuid)`.
    ```kotlin
    // feature/live-workout/.../mvi/handler/CommonHandler.kt:111-125
    private suspend fun createSession(trainingUuid: String?): String? {
        if (trainingUuid.isNullOrBlank()) return null
        return try {
            val session = interactor.startSession(trainingUuid)
            session.uuid
        } catch (e: Exception) {
            null
        }
    }
    ```
6.  **Persistence**: `LiveWorkoutInteractorImpl.kt:127-142` calls `sessionRepository.startSessionWithExercises`.

**Route Arguments (`Screen.LiveWorkout`)**
-   `sessionUuid: String?`: Non-null for resuming an existing draft.
-   `trainingUuid: String?`: Non-null for starting a fresh session from a Training template.
-   **Implication**: Current logic returns `null` if both are missing. Quick start would need to handle a "fresh with no training" case, either by allowing `trainingUuid` to be null or by creating a "Draft Training" row before navigation.

**Earliest persistence**
-   **On screen open**: The session row is persisted immediately during the `Action.Common.Init` phase (before the user sees the first exercise).
-   **Implication**: Abandoned Quick start sessions (opened but immediately left) will leave orphan rows in the DB unless explicitly cleaned up.

**Existing no-Training path**
-   None. `createSession` explicitly returns `null` if `trainingUuid` is missing (`CommonHandler.kt:112`).

## 2. Domain model: Session ↔ Training ↔ Exercise relationships

**Relevant Room Entities**
-   **`SessionEntity`** (`core/database/.../session/SessionEntity.kt:33-46`):
    -   `uuid: Uuid` (PK)
    -   `trainingUuid: Uuid` (**Non-nullable**, FK to `training_table`)
    -   `state: SessionStateEntity` (`IN_PROGRESS` | `FINISHED`)
    -   `startedAt: Long`, `finishedAt: Long?`
-   **`TrainingEntity`** (`core/database/.../training/TrainingEntity.kt:18-32`):
    -   `uuid: Uuid` (PK), `name: String`, `isAdhoc: Boolean`.
-   **`TrainingExerciseEntity`** (`core/database/.../training/TrainingExerciseEntity.kt:29-41`):
    -   `trainingUuid: Uuid`, `exerciseUuid: Uuid`, `planSets: String?` (JSON blob).

**Foreign Key Constraint**
-   `SessionEntity` has a non-nullable `training_uuid` with `onDelete = ForeignKey.CASCADE` (`SessionEntity.kt:16-21`).
-   **Implication**: A session **cannot exist** in the DB without a corresponding row in `training_table`. Quick start must create a Training row (likely marked `isAdhoc = true`) either at the start or during the session.

**Plan Sets JSON Blob**
-   Structure: `List<PlanSetDataModel>` (`PlanSetDataModel.kt:8-12`).
    ```kotlin
    data class PlanSetDataModel(
        val weight: Double?,
        val reps: Int,
        val type: SetTypeDataModel, // WARMUP, WORK, FAILURE, DROP
    )
    ```
-   **Null vs Empty**: `plan_sets` is nullable. In `TrainingExerciseEntity`, it is the template plan. In `ExerciseEntity.lastAdhocSets`, it is the most recently logged plan for ad-hoc sessions.

**`exercise.last_adhoc_sets`**
-   Written in `ExerciseRepositoryImpl.kt:114-122` via `setAdhocPlan`.
-   Read in `LiveWorkoutInteractorImpl.kt:65-72` as a fallback when a training-specific plan is missing for an ad-hoc session.
-   **Role**: Persists the user's "last used" weight/reps for an exercise when that exercise was trained outside of a fixed Training plan.

## 3. "Ad-hoc" concept in the current codebase

**Search findings**
-   `is_adhoc` on `TrainingEntity`: Distinguishes "Library" trainings from single-use or "Track now" trainings (`TrainingEntity.kt:28`).
-   `feature_home_recent_adhoc_label`: Used to label sessions in history that don't have a library training name (`strings.xml`).
-   `ExerciseInteractorImpl.startTrackNowSession`: Creates a new Training row with `isAdhoc = true` for a single exercise (`ExerciseInteractorImpl.kt:127-142`).

**Current Definition**
"Ad-hoc" currently means **(a) session bound to a Training row where `is_adhoc = true`**.
-   These trainings are typically created on-the-fly when a user clicks "Track now" from an Exercise Detail screen.
-   Evidence: `ExerciseInteractorImpl.kt:133` explicitly sets `isAdhoc = true` when creating a session for a single exercise.
-   **Implication**: Quick start fits the "Ad-hoc" category but extends it from "One exercise ad-hoc" to "Multi-exercise, no-name-yet ad-hoc".

## 4. Exercise creation flow

**Entry Points**
-   **All Exercises screen**: Clicking "Add" dispatches navigation to `Screen.Exercise(uuid = null)`.
-   **Exercise screen**: When `uuid == null`, it enters `Mode.Edit(isCreate = true)` (`ExerciseStore.kt:142`).

**Required Fields**
-   `name`: Checked in `ClickHandler.kt:211`. If blank, save is blocked and `nameError` is set to `true`.
-   `type`: Defaults to `WEIGHTED` (`ExerciseStore.kt:146`).

**Reactive Observation**
-   Exercises are observed via `ExerciseRepository.exercises` which returns a `Flow<PagingData<ExerciseDataModel>>` (`ExerciseRepositoryImpl.kt:50-55`).
-   Live workout loads exercises **one-shot** at session start via `SessionRepository.getDetail` and does not currently observe the global exercise list for additions during the session.

**Existing UX patterns**
-   **Inline creation**: `TagPickerInline.kt:85-90` shows a "Create '...'" button when a search query doesn't match existing tags.
    ```kotlin
    if (canCreate) {
        AppButton.Tertiary(
            text = stringResource(R.string.feature_exercise_edit_tag_create_format, searchQuery),
            onClick = { onTagCreate(searchQuery) },
            // ...
        )
    }
    ```
-   **Implication**: A similar pattern could be used in the exercise picker during Quick start.

## 5. Session lifecycle and draft persistence

**Persistence during workout**
-   **Every set**: `ClickHandler.kt:98-123` in `feature/live-workout` calls `interactor.upsertSet` immediately when a set is marked done.
-   **Draft sets**: Unfinished inputs are held in `Store.State.setDrafts` (`LiveWorkoutStore.kt:55`) and are **not** persisted to the DB until "Mark Done" is tapped.

**Finish Logic**
-   `SessionDao.finishSession` updates the row to `state = 'FINISHED'` and sets `finished_at` (`SessionDao.kt:182-191`).

**App background / Kill / Resume**
-   **Resume**: Home screen observes the active session via `SessionRepository.observeActiveSessionWithStats` (`HomeInteractorImpl.kt:41`).
-   **Banner**: `HomeScreen.kt:184` shows an "Active workout" banner if a session exists with `state = 'IN_PROGRESS'`.
-   **Implication**: Quick start sessions will automatically appear on Home for resume if the app is killed.

**Cleanup / Abandonment**
-   **Manual**: User can "Cancel session" (`ClickHandler.kt:254`) which calls `interactor.cancelSession` → `sessionRepository.deleteSession`.
-   **Automatic**: None. No `WorkManager` or startup hooks exist to delete stale `IN_PROGRESS` sessions.
-   **Implication**: If a user starts a Quick start and never finishes or cancels it, that `IN_PROGRESS` session (and its ad-hoc training row) stays in the DB forever.

## 6. PR snapshot mechanics

**PR Snapshot Loading**
-   **Location**: `LiveWorkoutInteractorImpl.kt:81-87`.
-   **Behavior**: Collected **once** via `firstOrNull()` at session start to freeze the baseline.
    ```kotlin
    val preSessionPrs = personalRecordRepository
        .observePersonalRecords(uuidsByType)
        .firstOrNull()
        .orEmpty()
    ```

**New Exercise Behavior**
-   Currently, adding an exercise during a session is **not supported**. The `Action.Click` surface in `LiveWorkoutStore.kt` has no "Add Exercise" variant.
-   **If added**: The `preSessionPrSnapshot` in `State` is a `Map<String, PrSnapshotItem>`. If a key is missing, `PrComparator.beats` treats it as `hasBaseline = false`.
    ```kotlin
    // core/exercise/.../sets/PrComparator.kt:66
    if (!hasBaseline) return candidateWeight > 0.0 || candidate.reps > 0
    ```
-   **Result**: For a brand-new exercise with no history, the **first set logged** with any weight/reps will be marked as a Personal Record (PR).

## 9. Schema version and migration baseline

-   **Current Version**: `5` (`AppDatabase.kt:33`).
-   **Existing Migrations**:
    -   v1 → v2: Destructive (pre-release).
    -   v2 → v3: Destructive (pre-release).
    -   v3 → v4: Destructive (pre-release).
    -   v4 → v5: Destructive (pre-release).
-   **Migration Policy**: From v5 onward, destructive migrations are forbidden. Explicit `Migration` objects must be registered in `CoreDatabaseModule.kt` (`architecture.md:368-380`).
-   **Schema Support**: The existing schema **does not** support a session without a Training plan due to the non-nullable FK on `session_table.training_uuid`. A Quick start session must either create a dummy Training row immediately or the schema must be bumped to v6 to make `training_uuid` nullable.

## 10. Open ambiguities

1.  **Session without Training**: The non-nullable `training_uuid` in `SessionEntity` contradicts the "no preconfigured Training" requirement if "no preconfigured" means "no DB row".
2.  **Exercise Creation during Session**: Current `LiveWorkoutInteractor` only loads the session structure once. If an exercise is created and added during the session, the interactor and store need to handle a dynamic, observable list of performed exercises.
3.  **PR Snapshot and New Exercises**: Since the PR snapshot is frozen at session start, a brand-new exercise created *during* the session won't have a snapshot entry. While `PrComparator` handles this (`hasBaseline = false`), the `LiveWorkoutStore.State.preSessionPrSnapshot` map needs to be mutable or updateable during the session.
